## 1. “更复杂的深度匹配模型难以直接用于全库召回”是什么意思？

这句话的核心含义是：

> **如果一个召回模型需要对“用户-商品”进行复杂交叉计算，那么它很难在在线阶段对全量商品逐个打分并取 TopK。**

推荐系统里的召回可以简化为：

```text
给定用户 u，从全量商品库 I 中找出最可能感兴趣的 TopK 商品。
```

如果商品库有 1 亿个商品，理论上最直接的做法是：

\[
score(u, i_1), score(u, i_2), ..., score(u, i_N)
\]

然后排序取 TopK。

问题在于，**复杂深度匹配模型的单次打分成本较高**，如果对全库每个商品都跑一遍，在线延迟和算力成本通常不可接受。

---

# 2. 为什么双塔内积适合全库召回？

典型 u2i 双塔模型是：

\[
score(u,i)=\langle f_u(u), f_i(i) \rangle
\]

它的关键优势是：**用户侧和商品侧可以解耦计算**。

## 2.1 商品向量可以提前离线算好

商品塔输出：

\[
v_i=f_i(i)
\]

所有商品的向量可以离线提前算好，并建立向量索引：

```text
item_1 → embedding_1
item_2 → embedding_2
...
item_N → embedding_N
```

在线时只需要计算用户向量：

\[
v_u=f_u(u)
\]

然后在向量索引中做近似最近邻检索：

\[
TopK_i \langle v_u, v_i \rangle
\]

这就可以用 Faiss、HNSW、ScaNN、Milvus、Vearch 等 ANN/MIPS 检索框架完成。

---

## 2.2 复杂深度匹配模型为什么难？

例如一个更强的交叉模型可能是：

\[
score(u,i)=DNN([u, i, u \times i, attention(u\_history, i)])
\]

这种模型会显式计算：

- 用户历史行为与当前商品的 attention；
- 用户特征与商品类目、品牌、价格的交叉；
- 用户序列与商品语义的深度匹配；
- 多层 MLP、Transformer 或 Cross Network。

这类模型的问题是：**商品侧表示依赖当前用户，无法提前完全离线计算**。

举例：

```text
用户 A 看过手机、耳机、电脑
用户 B 看过奶粉、纸尿裤、童装

同一个商品“iPhone 保护壳”，对用户 A 和用户 B 的 attention 结果不同。
```

因此商品 embedding 不是固定的，或者最终打分不是简单内积，导致无法直接使用标准 ANN 做全库检索。

---

## 2.3 一个直观对比

| 模型类型 | 打分形式 | 商品向量能否离线预计算 | 是否适合全库召回 |
|---|---|---:|---:|
| **双塔内积** | \(\langle u, i\rangle\) | 可以 | 适合 |
| **双塔 cosine** | \(cos(u,i)\) | 可以 | 适合 |
| **双塔 + ANN** | 向量近邻搜索 | 可以 | 适合 |
| **Cross Encoder** | DNN(user, item) | 很难 | 不适合直接全库 |
| **DIN/DIEN 类 attention 模型** | attention(user history, item) | 很难 | 通常用于排序/粗排 |
| **Transformer 交叉匹配** | 用户和商品 token 交叉编码 | 很难 | 通常用于重排 |

所以这句话不是说复杂模型不能用于推荐，而是说：

> **复杂深度匹配模型通常不能直接对全量商品做在线打分，只适合在召回之后的小候选集上做粗排、精排或重排。**

---

# 3. 全库召回的好处是什么？

**全库召回** 指的是召回阶段理论上可以从整个商品库、内容库或广告库中检索候选，而不是只从某个预先筛选的小池子里挑选。

它的主要价值包括以下几类。

---

## 3.1 覆盖面更大，减少候选池偏置

如果只从固定候选池召回，例如：

```text
热门商品池
类目商品池
运营商品池
用户历史相似商品池
```

系统容易产生偏置：

- 越热门的商品越容易被推荐；
- 新品和长尾商品很难进入推荐链路；
- 用户兴趣容易被历史行为锁死；
- 召回结果缺少探索性。

全库召回可以让所有商品都有机会被检索到，至少理论上降低了候选池带来的偏置。

---

## 3.2 更容易召回长尾和新品

对于电商场景，全量商品规模很大：

```text
头部商品：曝光多、点击多、训练充分
长尾商品：曝光少、点击少、训练不足
新品商品：刚上架，缺少行为数据
```

全库召回如果配合内容特征、类目特征、图文语义特征，可以更好地支持：

- 新品冷启动；
- 长尾商品曝光；
- 非热门但高度匹配的商品召回；
- 个性化供给挖掘。

---

## 3.3 更接近“全局最优 TopK”

如果只在局部候选池里召回，系统得到的是：

\[
TopK(i \in candidate\ pool)
\]

而不是：

\[
TopK(i \in all\ items)
\]

前者可能漏掉真正最适合用户的商品。

全库召回的目标是更接近：

```text
在全量商品中找到用户最可能点击、购买、停留或转化的商品。
```

这对推荐系统的上限很重要。

---

## 3.4 简化多路召回的候选依赖

传统推荐系统常见多路召回：

```text
热门召回
协同过滤召回
类目召回
品牌召回
向量召回
图召回
运营召回
```

多路召回虽然稳定，但会带来：

- 通道之间重复率高；
- 通道权重难调；
- 召回池结构复杂；
- 候选来源难统一评估；
- 新策略上线成本高。

如果全库召回能力足够强，可以减少对大量启发式召回通道的依赖。

---

## 3.5 有利于统一建模和端到端优化

全库召回可以把召回问题统一成：

\[
P(item \mid user, context)
\]

或者：

\[
score(user, item)
\]

从而更适合和后续排序目标对齐，例如：

- CTR；
- CVR；
- GMV；
- 用户长期价值；
- 内容消费时长；
- 多目标收益。

---

# 4. 目前业界常见的全库召回方法

业界全库召回大致可以分为以下几类。

---

# 4.1 双塔向量召回：Two-Tower + ANN

这是目前最主流的工业方案之一。

## 原理

分别学习用户向量和商品向量：

\[
v_u=f_u(user)
\]

\[
v_i=f_i(item)
\]

然后用内积、cosine 或其他向量相似度做召回：

\[
score(u,i)=v_u^\top v_i
\]

## 在线流程

```text
离线：
1. 训练双塔模型
2. 计算全量商品 embedding
3. 构建 ANN 向量索引

在线：
1. 计算用户 embedding
2. 在 ANN 索引中检索 TopK 商品
3. 返回召回候选
```

## 常见 ANN/MIPS 技术

| 方法 | 说明 |
|---|---|
| **HNSW** | 基于图的近似最近邻搜索，召回率高，延迟低 |
| **IVF** | 倒排文件索引，先聚类再局部检索 |
| **PQ/OPQ** | 向量量化压缩，降低内存和计算 |
| **LSH** | 局部敏感哈希，适合近似相似搜索 |
| **ScaNN** | Google 提出的向量检索方案 |
| **Faiss** | Meta 开源向量检索库，工业使用广泛 |
| **Annoy/NMSLIB** | 常见近似向量检索工具 |
| **Milvus/Vearch/向量数据库** | 工程化向量检索系统 |

## 优点

- **在线效率高**；
- **工程成熟**；
- **可扩展到亿级、十亿级商品库**；
- **商品向量可离线更新**；
- **适合高并发低延迟场景**。

## 缺点

- 表达能力受限于向量内积；
- 用户和商品交叉特征弱；
- 单向量难表示多兴趣；
- 对冷启动和复杂意图建模有限；
- 容易受到 embedding 空间结构限制。

---

# 4.2 多兴趣向量召回：MIND、ComiRec、MIP 等

这是对单向量双塔的增强。

## 原理

一个用户不只生成一个向量，而是生成多个兴趣向量：

\[
v_u^1, v_u^2, ..., v_u^m
\]

每个向量代表一种兴趣：

```text
兴趣 1：手机数码
兴趣 2：运动户外
兴趣 3：食品生鲜
兴趣 4：家居日用
```

召回时每个兴趣向量都去 ANN 索引中检索商品，然后合并去重。

## 在线流程

```text
1. 根据用户行为序列生成多个兴趣向量
2. 每个兴趣向量检索 TopK 商品
3. 合并多个召回结果
4. 去重、截断、简单打分融合
```

## 优点

- **缓解单向量兴趣平均问题**；
- 对多兴趣用户更友好；
- 可以提升召回多样性；
- 仍然兼容 ANN 检索，在线效率较高。

## 缺点

- 多个兴趣向量会增加检索次数；
- 兴趣数量难选择；
- 多兴趣之间可能重复召回；
- 最终打分仍多为内积，相比交叉模型表达能力仍有限。

---

# 4.3 TDM：Tree-based Deep Model

TDM 是用树结构替代直接全库打分。

## 原理

将所有商品组织成一棵树：

```text
Root
├── 节点 A
│   ├── 节点 A1
│   │   ├── 商品 1
│   │   └── 商品 2
│   └── 节点 A2
└── 节点 B
```

叶子节点是商品，内部节点是一组商品集合。

模型学习：

\[
score(u, node)
\]

在线从根节点开始逐层向下搜索，最后找到叶子商品。

## 在线复杂度

如果：

- 树高为 \(H\)；
- 每个节点分叉数为 \(K\)；
- beam size 为 \(B\)；

则在线打分量约为：

\[
O(B \cdot K \cdot H)
\]

远低于全库 \(O(N)\) 打分。

## 优点

- 可以使用比双塔内积更复杂的 user-node 匹配模型；
- 检索复杂度从全库扫描降低到树搜索；
- 可兼顾效率和表达能力；
- Beam Search 可保留多个兴趣方向。

## 缺点

- 树结构质量决定召回上限；
- 高层路径一旦选错，子树商品全部丢失；
- 训练和在线一致性复杂；
- 树更新、商品迁移、节点特征维护成本较高。

---

# 4.4 Deep Retrieval / Learning to Retrieve

Deep Retrieval 类方法希望直接学习一个“检索路径”或“离散编码”，让模型通过多步选择定位商品。

## 核心思路

不再显式对全量商品打分，而是把商品映射到某种可搜索结构中，例如：

```text
商品 → 离散编码
商品 → 多级类别路径
商品 → 语义 token 序列
```

在线时模型根据用户生成路径或编码，再映射到商品集合。

例如：

\[
user \rightarrow code_1 \rightarrow code_2 \rightarrow code_3 \rightarrow items
\]

## 典型流程

```text
离线：
1. 为商品学习离散编码或层级结构
2. 训练用户到编码路径的模型
3. 建立编码到商品的倒排表

在线：
1. 输入用户特征
2. 模型预测若干高概率编码路径
3. 根据编码路径查找对应商品
4. 合并、排序、截断
```

## 优点

- 规避全库逐 item 打分；
- 比单向量内积表达能力更强；
- 可以学习非欧式空间的检索结构；
- 适合大规模候选生成。

## 缺点

- 商品编码学习难；
- 编码冲突和编码稀疏会影响效果；
- 训练目标和最终召回目标对齐较难；
- 在线需要控制路径数，否则候选爆炸；
- 工程复杂度高于双塔 ANN。

---

# 4.5 生成式召回：Generative Retrieval

近年来比较热门的一类方法是 **生成式推荐 / 生成式召回**。

## 原理

把 item 表示为一个语义 ID 或 token 序列：

```text
item_123 → [12, 37, 405, 8]
item_456 → [12, 38, 291, 5]
```

模型输入用户行为序列，然后自回归生成用户可能感兴趣的 item ID：

\[
P(code_1, code_2, ..., code_m \mid user)
\]

最终通过生成的 code 找到具体商品。

## 在线流程

```text
1. 输入用户历史行为和上下文
2. 生成模型输出多个 item semantic ID
3. 将 semantic ID 映射回商品
4. 做合法性校验、去重和截断
```

## 优点

- 可以摆脱传统向量内积检索框架；
- 表达能力强，适合序列建模；
- 能学习 item 之间的语义层级；
- 有机会统一召回、排序和生成任务。

## 缺点

- 商品 ID 编码设计非常关键；
- 生成非法 ID、重复 ID 是常见问题；
- 大规模商品动态更新较复杂；
- 在线推理成本可能较高；
- 工程成熟度整体低于双塔 ANN。

---

# 4.6 图召回：Graph-based Retrieval

图召回也是工业界常用的大规模召回方式。

## 原理

构建用户、商品、类目、品牌、店铺等异构图：

```text
用户 — 点击 — 商品
用户 — 购买 — 商品
商品 — 属于 — 类目
商品 — 来自 — 店铺
商品 — 相似 — 商品
```

然后通过图算法召回候选：

- random walk；
- Personalized PageRank；
- item-item graph；
- user-item graph；
- GraphSAGE；
- PinSAGE；
- GNN embedding。

## 在线方式

常见有两种：

### 方式一：图遍历召回

```text
1. 从用户最近行为商品出发
2. 在图上扩展邻居
3. 根据边权、路径权重、时间衰减打分
4. 返回候选商品
```

### 方式二：图 embedding + ANN

```text
1. 离线训练商品或用户图 embedding
2. 建立商品向量索引
3. 在线用用户 embedding 检索商品
```

## 优点

- 能利用协同行为和关系网络；
- 对相似商品、互补商品召回有效；
- 可解释性较好；
- 适合多关系、多实体场景。

## 缺点

- 图更新成本较高；
- 容易偏向热门节点；
- 在线多跳遍历可能延迟不可控；
- 图 embedding 最终仍可能退化成向量召回。

---

# 4.7 稀疏特征倒排召回：Inverted Index / Sparse Retrieval

这类方法在搜索、广告、推荐中都很常见。

## 原理

为商品建立倒排索引：

```text
类目=手机 → item_1, item_2, item_3
品牌=Apple → item_1, item_5
关键词=无线耳机 → item_8, item_9
价格带=1000-2000 → item_20, item_21
```

在线根据用户画像、query、上下文触发倒排检索。

## 常见形式

- query-item BM25；
- tag-item 倒排；
- category-item 倒排；
- user profile tag 匹配；
- sparse embedding 检索；
- SPLADE / learned sparse retrieval。

## 优点

- 在线效率极高；
- 可解释性强；
- 对搜索推荐融合友好；
- 易加入业务规则和过滤条件。

## 缺点

- 语义泛化能力弱；
- 依赖人工特征或文本质量；
- 个性化能力不如深度模型；
- 难捕捉复杂兴趣迁移。

---

# 4.8 矩阵分解与协同过滤召回

这是经典召回方法。

## 原理

根据用户-商品交互矩阵学习隐向量：

\[
R \approx U V^\top
\]

用户和商品分别有隐向量，召回时取内积最高的商品。

常见方法包括：

- Matrix Factorization；
- ALS；
- BPR；
- ItemCF；
- UserCF；
- Swing；
- SLIM；
- EASE。

## 优点

- 简单稳定；
- 对协同行为信号利用充分；
- 工业可解释性较好；
- 可作为召回基线或补充通道。

## 缺点

- 冷启动弱；
- 难利用复杂内容特征；
- 对稀疏行为敏感；
- 通常个性化表达能力有限。

---

# 4.9 暴力全库打分：Brute-force Scoring

在商品规模较小或算力充足时，也可以直接全库打分。

## 适用场景

例如：

- 商品库只有几万到几十万；
- 召回频率不高；
- 可以 GPU 批量计算；
- 模型较轻；
- 离线推荐，不要求毫秒级响应。

## 优点

- 最接近精确 TopK；
- 不受 ANN 近似误差影响；
- 实现逻辑简单；
- 可使用更灵活的打分函数。

## 缺点

- 难扩展到千万级、亿级商品；
- 在线成本高；
- 延迟不稳定；
- 高并发场景不可持续。

---

# 5. 不同全库召回方法对比

| 方法 | 是否真正覆盖全库 | 表达能力 | 在线效率 | 工程成熟度 | 主要优点 | 主要缺点 |
|---|---:|---:|---:|---:|---|---|
| **双塔 + ANN** | 高 | 中 | 高 | 高 | 高效、成熟、可扩展 | 受内积表达限制 |
| **多兴趣向量召回** | 高 | 中高 | 较高 | 中高 | 解决多兴趣问题 | 检索次数增加 |
| **TDM** | 高 | 高 | 中高 | 中 | 可用复杂匹配模型 | 树结构依赖强 |
| **Deep Retrieval** | 高 | 高 | 中高 | 中 | 学习离散检索结构 | 训练和编码复杂 |
| **生成式召回** | 高 | 高 | 中 | 中低 | 表达能力强、范式新 | 工程成熟度不足 |
| **图召回** | 中高 | 中高 | 中 | 中高 | 利用关系网络 | 热门偏置、更新复杂 |
| **稀疏倒排召回** | 中高 | 中 | 高 | 高 | 快、稳、可解释 | 语义泛化弱 |
| **协同过滤召回** | 中高 | 中 | 高 | 高 | 简单稳定 | 冷启动弱 |
| **暴力全库打分** | 最高 | 最高 | 低 | 中 | 精确、灵活 | 成本极高 |

# 6. 关键结论

1. **“复杂深度匹配模型难以直接用于全库召回”**，主要是因为它们需要在线对每个 user-item pair 做交叉计算，无法像双塔模型一样预先计算商品向量并建立 ANN 索引。

2. **全库召回的最大好处** 是候选覆盖面大、减少固定候选池偏置，更有机会找到全局最优商品，并提升新品、长尾和多兴趣召回能力。

3. **当前工业界主流全库召回方案** 仍然是 **双塔向量召回 + ANN/MIPS**，因为它在效果、延迟、成本、工程成熟度之间平衡最好。

4. 为突破双塔内积瓶颈，业界探索了 **多兴趣召回、TDM、Deep Retrieval、生成式召回、图召回、稀疏神经检索** 等方法。

5. 实际系统中通常不是单一方法，而是组合式架构：

```text
双塔 ANN 召回
+ 多兴趣召回
+ 图召回
+ 稀疏倒排召回
+ 热门/新品/运营召回
→ 粗排
→ 精排
→ 重排
```

**下一步建议**：如果你关注的是“如何突破双塔召回上限”，可以重点比较三条路线：  
**多兴趣向量召回**、**树/离散结构召回 TDM/Deep Retrieval**、**生成式召回**。这三类方法分别代表了“扩展用户表示”“改变检索结构”“改变 item 表达方式”三种方向。
